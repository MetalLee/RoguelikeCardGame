# 技术需求

状态：草案  
上级索引：[[design/README|Design Knowledge Base]]

## 目的

定义第一版 MVP 和后续开发的技术边界、架构原则、工具需求和验证方式。

## 当前实现状态提示

截至 2026-06-21，项目已进入 `combat-rework-beat` 三拍拆招重构切片。Godot 4.6.x .NET / C# 旧版 MVP 原型可作为工程基础和历史原型，但旧“连锁层数、技能牌、卡牌包奖励、3 / 5 / 8 阈值”方向已退出主流程；当前技术需求以三拍拆招、无色彩能、终结槽和武器卡池为准，五色附魔延后。

## 技术原则

- Godot 工程根目录固定为 `game/`。
- 玩法规则层与 Godot 表现层分离。
- C# 规则逻辑不要依赖 Godot 节点树。
- 卡牌、武器、魔物、遗物、遭遇、奖励、色彩和本地化优先数据驱动，不要把内容数值写死在场景里。
- 表现层只读取规则层结果和结构化日志，不自行结算伤害、彩能、色彩、费用、防御或胜负。
- 每次新增或修改代码、数据时，同步补充可运行校验、测试或最小验证方式。

## 第一版 MVP 必须支持

- 主角「左」。
- 起始武器池：左轮剑、机械臂。
- 起始主副武器选择：玩家可自由选择左轮剑或机械臂作为主手，另一个作为副手。
- 起始卡组自动生成：每把武器固定初始池为 4 张，其中 3 张行动牌 + 1 张终结牌；主手和副手各 4 张全部加入，组成 8 张初始卡组。
- 行动牌和终结牌两类卡牌；不再保留技能牌类型。
- 玩家拍位基础长度为 3；行动点只作为旧原型兼容或后续扩展字段。
- 彩能基础上限 6，跨轮不保留。
- 行动牌以动作串参与对撞，成功造成伤害、有效格挡或成功闪避时生成无色彩能。
- 5 种色彩红、黄、蓝、绿、紫延后到五色阶段启用。
- 终结牌按自身数据消耗固定数量、X 数量或全部无色彩能，并由玩家从终结槽手动释放。
- 普通战斗后奖励：当前为武器卡牌三选一；后续五色阶段再恢复随机色彩碎片。
- 精英战斗后额外普通遗物。
- Boss 击败后通关结算 + 简化 Boss 奖励。
- 每场战斗开始时玩家生命回满；生命归零立即结束本次 MVP Run。
- 3 场固定遭遇，魔物语境统一到黑塔世界观。

## 规则层需求

规则层应至少包含：

- `WeaponDefinition`：武器 ID、初始卡池、奖励卡池、武器标签。
- `CardDefinition`：所属武器、卡牌类型、拍位占用、动作串、目标、彩能生成 / 消耗、效果 DSL；色彩互动为后续五色字段。
- `BeatCombatState` / `CombatState`：生命、玩家拍位、魔物拍位、目标锁定、无色彩能槽、终结槽、牌区、魔物、日志。
- `ColorDefinition` 或等价枚举：后续五色阶段启用红、黄、蓝、绿、紫与无色。
- `ColorShard` / 附魔状态：后续五色阶段记录行动牌是否已附魔及其颜色。
- `RunState`：当前武器、主卡组、遗物、当前节点、Boss 奖励状态；后续五色阶段加入附魔状态。
- `CombatLogEvent`：记录拍位放置、目标锁定、动作对撞、彩能变化、终结释放、防御 / 格挡 / 闪避变化、抽弃牌、魔物意图和触发效果。

## 表现层需求

- 支持起始主副武器选择界面；起始卡牌由系统自动生成，选牌界面不进入当前 MVP 主流程。
- 支持 6 格基础无色彩能槽；后续五色阶段再显示每点彩能颜色。
- 支持玩家拍位、魔物拍位、目标指向箭头、对撞目标和空门目标反馈。
- 后续五色阶段支持行动牌附魔标记。
- 支持终结牌预览：消耗彩能数量、预计伤害或资源收益、结算后剩余彩能；颜色构成后续接回。
- 后续五色阶段支持色彩碎片奖励和附魔界面。
- 支持武器卡牌三选一奖励。
- Boss 战后奖励结算延后；当前 Boss 胜利直接进入通关结算。
- 动画、音效和屏幕反馈应由结算事件驱动，避免表现层与规则层各算一套结果。

## 数据与工具需求

- JSON + JSON Schema 仍作为首版内容数据格式。
- 数据校验必须检查内容 ID 唯一、引用存在、武器固定初始池数量与 4 行动 / 2 终结结构、卡牌类型、动作串、彩能消耗、奖励候选和本地化文本；五色阶段再检查色彩引用。
- 战斗沙盒需要支持：选择固定遭遇、选择起始主副武器或指定调试卡组、设置随机种子、设置初始生命 / 拍位长度 / 彩能、指定魔物拍位序列、加入指定遗物；五色阶段再支持指定色彩碎片。
- 试玩指标导出需要记录拍位放置、目标锁定、动作对撞、彩能生成 / 消耗、终结牌释放和奖励选择；五色阶段再记录色彩与附魔。

## 项目目录

根据 [[design/08_governance/2026-06-04_keep_godot_project_in_same_repo_game_subdir|Godot 工程与知识库同仓库并放入 game 子目录]]，第一版 MVP 阶段 Godot 工程与知识库保留在同一个 Git 仓库中，但 Godot 工程根目录固定为 `game/`。知识库根目录继续维护 `design/`、`inspiration/`、`insight/` 等文档目录，游戏实现相关文件放在 `game/` 下。

## 验证要求

- 数据校验通过。
- 规则层单元测试覆盖主副武器选择后的自动起始牌组、行动牌生成彩能、附魔、终结牌消耗彩能、五色附加效果、战后奖励、精英遗物、Boss 奖励和失败结算。
- Godot headless 或可用启动验证通过。
- 若当前实现仍未迁移，应在验证结果中明确“旧 MVP 通过，新设计未实现”。
