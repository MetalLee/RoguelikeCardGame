# 存档、配置与平台

状态：草案  
上级索引：[[design/README|Design Knowledge Base]]

## 目的

定义存档、设置、平台目标和用户配置要求，确保玩家进度可靠、试玩数据可复盘，并让 Godot + C# 技术栈服务当前制作阶段。

## 平台目标

首版开发和试玩目标为 PC 桌面平台，Windows 优先；后续再考虑 macOS、Linux 支持。Web、移动和主机不进入第一版 MVP 范围。

## 存档目标

存档系统服务三个目标：

- 可靠恢复：玩家关闭游戏后能回到可理解的 Run 状态。
- 调试复现：开发者能通过存档、随机种子和战斗日志复盘问题。
- 版本兼容：内容和规则变化后，旧存档不会无提示崩溃。

## 存档范围

### 第一版 MVP

MVP 至少保存：

- 当前 Run ID 和随机种子。
- 当前节点序号和节点 ID。
- 当前主手武器和副手武器。
- 自动起始牌组结果。
- 当前卡组列表；若同名牌可重复且附魔不同，需要保存卡牌实例 ID。
- 当前卡牌附魔状态。
- 当前遗物列表。
- 当前玩家生命上限与当前生命；即使 MVP 每战开始回满，也要保留字段。
- 已完成节点记录。
- 当前奖励选择状态，包括色彩碎片、候选卡牌、精英遗物和 Boss 简化奖励。
- MVP 通关或失败状态。
- 试玩统计摘要。
- 内容版本与存档版本。

MVP 只需要节点级自动存档：Run 开始、主副武器确认并生成初始牌组、战斗开始、战斗胜利、奖励确认、附魔确认、精英遗物获得、Boss 通关和失败结算时保存。战斗中逐动作恢复不是 MVP 必须项，留到后续阶段评估。

### 垂直切片以后

后续完整 Run 需要补充：

- 路线图状态和可见节点信息。
- 事件、商店、休息或升级节点状态。
- 货币、删牌、升级、治疗等经济状态。
- 局外解锁、图鉴、挑战等级和统计数据。
- 当前战斗中断恢复，必要时保存行动栈或战斗事件流。

## 存档格式

- 使用 Godot `user://` 目录保存玩家本地数据。
- 采用 JSON 或二进制外壳 + JSON payload 的版本化格式；首版优先 JSON，方便调试和 Codex 读取。
- 每个存档包含 `save_version`、`content_version`、`created_at`、`updated_at`。
- 写入采用临时文件 + 原子替换，避免写入中断导致主存档损坏。
- 保留最近一次自动备份；主存档读取失败时尝试读取备份。
- 存档读取失败时给出可读错误，并允许玩家重新开始 MVP Run。

## 版本与迁移

- `save_version` 表示存档结构版本。
- `content_version` 表示内容数据版本。
- 内容 ID 改名必须通过迁移表处理，不直接让旧存档引用失效。
- 2026-06-13 新设计替代旧连锁 / 技能牌 / 卡牌包结构，旧测试存档可破坏性重置，但需要在版本说明和变更日志中记录。
- 如果旧存档无法迁移，应给出明确说明，并在开发测试版本中输出详细日志。

## 设置需求

### 音频

- 主音量。
- 音乐音量。
- 音效音量。
- UI 操作音量。

音量分类需要支持 [[design/03_experience/02_audio_feedback|音频与反馈]] 中的操作确认、战斗结果、风险提示和彩能 / 色彩反馈。

### 画面与窗口

- 分辨率。
- 全屏 / 窗口 / 无边框窗口。
- 垂直同步。
- UI 缩放，至少为后续长文本和高分辨率适配保留接口。

### 动画与反馈

- 动画速度。
- 减少动画。
- 屏幕震动开关或强度。
- 伤害数字与战斗日志显示开关。

这些设置直接服务彩能生成、色彩附着、终结牌释放和魔物意图反馈，不能削弱规则可读性。

### 语言

首版开发语言和默认游戏文本为简体中文。所有运行时文本应通过本地化键读取，避免把中文文案写死在规则代码中。垂直切片或公开演示前再评估英文文本。

### 输入

首版输入目标为键鼠。必须支持：

- 鼠标选择卡牌、目标、奖励、附魔对象和 UI 按钮。
- 键盘基础快捷操作，例如结束回合、确认、取消、打开牌库或调试入口。
- 基础按键配置结构，MVP 可先提供默认键位，垂直切片后再提供完整自定义键位 UI。

手柄、触摸和移动端输入不进入第一版 MVP 范围。

## 配置保存

设置项与 Run 存档分离保存，避免玩家重开或删除 Run 时丢失音量、窗口和输入偏好。

配置文件至少包含：

- `config_version`
- 音频设置。
- 显示设置。
- 动画与反馈设置。
- 语言设置。
- 输入映射。

配置读取失败时应回退默认设置，并保留错误日志。

## 统计与隐私

第一版 MVP 使用本地试玩数据导出，不接入在线遥测后台。测试者数据应由开发者主动导出或收集，避免在核心循环尚未稳定前引入账号、网络、隐私合规和后台维护成本。

本地统计可保存：

- 总游玩局数。
- MVP 通关次数和失败次数。
- 最高单回合彩能数量。
- 最常附魔色彩。
- 最高单次终结牌伤害。
- 最近一次 Run 摘要。
- 调试构建中的详细战斗日志。

第一版 MVP 暂不需要在线遥测、云存档、账号或后台服务。公开测试前若重新评估在线遥测，必须另行确认数据范围、用户告知、隐私策略和开关。

## 平台适配准则

- 平台适配不早于核心循环验证。
- 任何平台目标都不能牺牲战斗 UI 可读性、卡牌文本可读性和彩能 / 色彩反馈。
- 分辨率适配应优先保证战斗界面、手牌区、魔物意图和彩能 UI。
- 移动端或手柄适配若导致交互结构大改，应作为单独里程碑处理。

## 验收标准

第一版 MVP 的存档、配置与平台需求达到以下标准即可进入内部测试：

- Windows 桌面可启动、可完成 1 次 MVP Run、可重开。
- 关闭并重启游戏后，能从最近节点继续或清楚开始新 Run。
- 设置项能保存并在重启后生效。
- 存档损坏时不导致游戏无法启动。
- 战斗日志和基础统计可导出。
- 不存在 Web、移动、主机等非首版平台的阻塞性工作。

## 已确认技术生产问题

首发平台、数据编辑偏好、战斗中断恢复粒度和在线遥测需求已在 [[inspiration/2026-06-03_06_technical_production_qa|06 Technical Production Q&A]] 中确认：Windows PC 为主，后续再考虑 macOS、Linux 支持；内容数据一致使用 JSON + JSON Schema；第一版 MVP 只需要节点级自动存档；只做本地试玩日志和统计导出。
