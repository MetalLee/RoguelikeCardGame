# 界面与交互

状态：草案  
上级索引：[[design/README|Design Knowledge Base]]

## 目的

定义玩家如何读取状态、理解意图、执行操作和复盘结果。Roguelike 卡牌游戏的 UI 是核心玩法的一部分。

## 核心界面

- 主菜单。
- 角色选择。
- 地图路线。
- 战斗界面。
- 奖励选择。
- 商店。
- 事件。
- 牌库查看。
- 遗物和状态查看。
- 胜利与失败结算。

## 战斗界面信息优先级

1. 玩家生命、资源、当前连锁层数、手牌。
2. 敌人生命、意图、状态。
3. 当前可执行行动。
4. 抽牌堆、弃牌堆、消耗堆。
5. 遗物、持续效果和战斗日志。

## 第一版 MVP 界面方向

第一版 MVP 的界面应服务“手绘漫画书风格”。战斗界面可以像一页动态漫画：战斗舞台使用清晰的漫画格或页边框组织角色、敌人与反馈，关键攻击和终结牌释放可以短暂切入冲击格、速度线、局部特写或拟声字，但这些演出不能移动核心 UI 的稳定位置。

MVP 战斗界面默认按 PC 16:9 横屏设计：玩家生命、防御和状态保持在左上稳定区；右上保留敌人 HUD 区域，用于在敌人悬停或单体目标箭头指向时显示敌人名称、生命、防御、状态和意图，默认不显示敌人信息；行动点与抽牌堆保持在左下；弃牌堆与结束回合按钮保持在右下；手牌位于底部中间，并以紧凑扇形叠放；当前连锁层数位于中上方，使用 8 格连锁点进度作为主要表达。

敌人立绘应作为透明舞台层融入战斗背景，不应被放入大面积黑色卡片或面板中。敌人状态信息可以使用轻量纸质状态条承载；敌人悬停、单体目标箭头指向、意图和受击反馈通过 HUD、箭头、卡牌有效目标蒙版和 VFX 表达。战斗中不使用点击敌人预先选取目标，也不显示独立的目标选中图标。

奖励、结算和牌库查看界面可以延续漫画书的页框、墨线和纸张纹理，但不应把核心选择做成纯插画。普通战斗奖励仍需清楚表达“先选卡牌包类型，再从 3 张同类型候选中选择 0-3 张加入卡组”的两段式流程。奖励包选择阶段的卡牌包应使用透明背景的手绘纸质包装图，直接叠在场景背景上；不得再包裹大面积黑色卡片或暗黑哥特底板。卡牌包的 hover 和点击命中应跟随 PNG 非透明区域，不能用覆盖整块版面或标签区域的透明矩形按钮代替真实卡包命中。跳过拿牌 / 进入下一战属于次级流程动作，应使用轻量纸质按钮，与战斗界面主操作按钮区分但保持可读。

## 第一版 MVP 战斗场景设计标准

当前战斗场景设计标准图：

![[design/03_experience/assets/battle_ui_effect_concept_2026-06-09_v2.png|第一版 MVP 战斗场景设计标准]]

该图作为后续 Godot 战斗界面、HUD 布局、角色 / 敌人舞台空间和手牌区域的当前对齐目标。正式运行时的生命、防御、行动点、连锁、意图、卡牌名称、费用和效果文本仍由 Godot UI 根据规则层与本地化数据绘制，不应把标准图中的文字直接烘焙为最终界面。

设计标准：

- 整体气氛采用明亮、温暖、可读的手绘漫画书风格，以纸张纹理、墨线、网点和轻量色块组织信息；默认 UI 不走暗黑哥特、恐怖或重黑装饰方向。
- 主角与敌人处于同一水平战斗线，比例清楚且留出足够垂直舞台空间，方便后续放入更高、更大的精英或 Boss 敌人。
- 左上角只显示玩家生命、防御和状态条，不显示主角头像；右上角保留敌人 HUD 区域，不显示敌人头像。该区域默认隐藏；鼠标悬停存活敌人，或拖拽 `TargetRule.SingleEnemy` 卡牌且箭头指向有效敌人时，才显示该敌人的名称、生命、防御、状态和意图。
- 左下角显示当前行动点与抽牌堆图标 / 数量；右下角显示弃牌堆图标 / 数量与结束回合按钮。
- 中间下方为手牌区，手牌使用竖版卡牌比例，紧凑叠放并以扇形展开，中心牌可以轻微抬高用于悬停或选中反馈。
- 第一版 MVP 的手牌出牌交互采用拖拽释放：悬停卡牌时仅抬升并置顶，不触发布牌；行动牌、技能牌、终结牌都需要按住拖拽后释放才会请求出牌。`TargetRule.Self`、`TargetRule.AllEnemies` 和 `TargetRule.None` 卡牌释放到主角、敌人和手牌之间的中场区域才可出牌；`TargetRule.SingleEnemy` 卡牌必须用漫画风箭头明确指向有效敌人目标后释放，不能只拖到中场区域出牌，也不能通过预先点击敌人来选定目标。表现层只负责 hover、drag、箭头、释放区、HUD focus 和卡牌有效目标蒙版，费用、连锁、目标合法性和胜负仍由规则层校验。
- 中上方连锁 UI 使用 `8` 格连锁点进度作为主要表达，例如 `连锁 1/8`；`3 / 5 / 8` 仍是机制阈值，但在 HUD 中只做弱提示，不再作为大号数字视觉焦点。

## 第一版 MVP 运行时资源接入

第一版 MVP 的 Godot 入口需要从 `game/data/presentation/` 读取表现 view 与 asset manifest，将背景、主角、敌人、卡牌模板、卡面插画、卡牌包、遗物图标和 UI 图标接入实际界面。资源文件只提供视觉层；卡牌名、费用、连锁需求、效果文本、生命、防御、意图和日志仍由 Godot UI 根据规则层和本地化数据绘制。

当前主菜单、战斗、奖励选择和结算流程可以先由程序化 UI 承载，但不得回退到无资源的纯占位界面。后续若拆分独立 `BattleScreen`、`RewardScreen` 或卡牌 / 敌人组件场景，应继续复用同一套 `presentation` 数据，不在场景节点里硬编码卡牌、敌人或奖励资源路径。

## 第一版 MVP 动效范围

第一版 MVP 动效采用规则日志驱动的轻量 Tween / VFX 方案：表现层读取新增的 `CombatLogEvent`，播放卡牌出手、敌人受击、防御闪光、连锁提升、3 / 5 / 8 阈值达成、终结牌冲击波、敌人行动、抽弃牌和死亡淡出等反馈。表现层不得自行推断伤害、连锁、费用、防御或胜负，只能根据规则层已经写入的结构化日志选择动画。

奖励界面第一版动效覆盖卡牌包选择入场、卡牌包打开、候选卡依次弹出和选卡确认反馈。卡牌包入场应是短促的淡入 / 弹出反馈，打开时可以叠加纸屑、连锁火花或冲击线 VFX，但不移动标题、说明、按钮等稳定 UI。该阶段不要求完整逐帧角色动作、骨骼动画或镜头时间轴；后续若增加主角 / 敌人动作，应继续由同一套事件驱动机制触发。

## 交互准则

- 卡牌目标、费用不足、无法行动原因必须即时反馈。
- 当前连锁层数、下一层变化和已满足的高连锁条件必须即时反馈。
- 连锁层数 UI 以 8 格连锁点进度为主，并在即将达到 3 / 5 / 8 机制阈值时提供弱提示、卡牌高亮或 VFX 反馈；HUD 中不应让 3 / 5 / 8 大号数字压过当前连锁进度。
- 连锁层数不设上限，UI 需要能显示超过 8 层后的当前层数，并避免让玩家误解 8 层是硬上限。
- 终结牌触发时需要清楚展示关键数值变化，例如伤害、倍率、暴击率、抽牌数、回费数或状态层数。
- 当卡牌因连锁层数满足额外效果时，卡牌本体需要有明显但不过度干扰的高亮状态。
- 终结牌需要预告结算后连锁层数是否归零；若存在保留效果，必须显示保留后的预计层数。
- 回合结束前需要提示连锁层数将归零，除非当前遗物或卡牌效果会保留部分或全部层数。
- 敌人意图和状态说明应优先通过悬停查看；点击敌人不得改变战斗目标选择状态。
- 关键决策需要确认，普通重复操作不应频繁打断。
- UI 文案优先清晰，少用只服务氛围的模糊表达。
