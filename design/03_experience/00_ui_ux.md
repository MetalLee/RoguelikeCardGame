# 界面与交互

状态：草案  
上级索引：[[design/README|Design Knowledge Base]]

## 目的

定义玩家如何读取状态、理解意图、执行操作和复盘结果。Roguelike 卡牌游戏的 UI 是核心玩法的一部分。

## 核心界面

- 主菜单。
- 起始武器选择。
- 地图路线。
- 战斗界面。
- 战后奖励与后续色彩附魔。
- 事件。
- 商店。
- 牌库查看。
- 遗物、色彩和状态查看。
- 胜利与失败结算。

## 战斗界面信息优先级

1. 玩家生命、当前无色彩能数量、手牌、玩家拍位、终结槽。
2. 魔物生命、魔物拍位、动作意图、状态。
3. 当前可执行编排、目标锁定状态与无法放入 / 无法瞄准原因。
4. 抽牌堆、弃牌堆、消耗堆。
5. 遗物、持续效果、色彩效果预览和战斗日志。

## 三拍拆招界面方向

当前重构方向来自 [[inspiration/2026-06-20_combat_identity_and_animation_brainstorm_qa|战斗特色化与动画爽感头脑风暴 Q&A]] 与 [[docs/superpowers/specs/2026-06-20-combat-rework-design|战斗重构设计：三拍拆招、成功产彩与终结分镜切入]]。第一版重构 MVP 的界面目标是让玩家一眼读懂“魔物将出什么动作、自己三拍准备怎么拆、哪些动作已经接住、哪里还有危险、哪里打出了空门，以及终结槽何时可以手动释放”。

战斗界面需要新增或强化以下表达：

- 玩家拍位：稳定显示 3 个玩家拍位，每个拍位展示已放入的行动牌、目标魔物拍位、动作串摘要和结算状态。
- 魔物动作序列：每个魔物公开本轮动作序列，以卡牌化动作块展示攻击、格挡、闪避等动作，并标明拍位顺序。
- 已接招状态：被玩家拍位锁定的魔物动作应显示“已接招”状态，明确该拍位会进入动作对撞。
- 危险状态：未被锁定且仍会生效的魔物动作应显示“危险”状态，帮助玩家判断剩余威胁。
- 空门状态：当某个魔物全部拍位都已被拆完，界面应提示该魔物进入“空门”状态，剩余玩家拍位可以直接攻击本体。
- 动作串展示：行动牌 hover、拖拽和放入拍位后，都需要展示该牌内部动作串，例如“格挡 -> 钝击”或“闪避 -> 斩击 x2”。
- 终结槽提示：终结槽固定显示当前放入的终结牌、所需彩能、当前无色彩能数量和可释放状态；当成功动作产彩并满足条件时，用高亮、短暂停顿或漫画分镜预备效果提示玩家可以手动释放。

### 三拍槽位瞄准交互

第一版可玩原型采用“两步式连续操作”：玩家先将行动牌拖入空的玩家拍位并松开鼠标，系统立即以该拍位作为临时锚点拉出箭头，等待玩家选择目标。若魔物仍有未锁定拍位，只能选择未锁定魔物拍作为对撞目标；当该魔物所有拍位都已锁定后，才允许从已填入行动牌的玩家拍位选择魔物本体。手牌不允许直接拖到魔物拍或魔物本体。若玩家在临时瞄准期间右键取消或点击空白区域，本次刚放入的行动牌退回手牌，该玩家拍位清空。

第一版重构 MVP 暂不展示色彩碎片附魔与五色终结转译。彩能 UI 可先以无色彩能计数和 6 格容量表达为主；后续恢复五色时，再把槽位颜色、附魔标记和终结转译预览接回界面。

## 第一版 MVP 界面方向

第一版 MVP 的界面应服务“手绘漫画书风格”和《剑与黑塔》的失色 / 夺色主题。战斗界面可以像一页动态漫画：战斗舞台使用清晰的漫画格或页边框组织角色、魔物与反馈，关键攻击和终结牌释放可以短暂切入冲击格、速度线、局部特写或拟声字，但这些演出不能移动核心 UI 的稳定位置。

后续 MVP 占位 UI、美术零件和战斗演出素材必须遵循 [[design/03_experience/01_visual_direction|视觉方向]] 中的“MVP 占位美术最高准则”。`design/03_experience/assets/` 下的黑白线稿资源是 UI 视觉承载的最高参考：界面应像在白纸漫画页上叠加可操作控件，而不是在彩色插画或暗色游戏 HUD 上叠加卡牌。

界面承载规则：

- 主菜单首屏保留主角「左」、标题《剑与黑塔》和“开始游戏”按钮；标题使用 `SourceHanSansSC-Heavy` 并保持第一视觉层级，开始按钮位于标题正下方。
- 开始游戏按钮使用白底、黑边、黑字的手绘平行四边形样式，不使用图标、暗色底板或金色按钮框。
- 默认底色使用白色、近白或浅灰纸面，关键区域用黑色墨线、漫画格边界、少量排线和轻纸纹分隔。
- 角色、魔物、卡牌插画、VFX 和图标应优先使用透明 PNG 分层，避免被固定黑色底板或彩色矩形背景锁死。
- 黑色块面可用于按钮描边、冲击格、Boss 剪影、爆炸重心和少数强调区域，但不应变成大面积暗色 UI 主题。
- 彩能槽、目标指向、后续附魔标记、色彩碎片和色彩核心允许使用机制色；颜色面积应服务机制编码，不能覆盖黑白线稿主体。
- UI 文本、数值、卡牌规则文本和本地化内容必须由 Godot UI 绘制，不应烘焙进 AI 生成图片。
- 战斗背景应保留高留白和低地平线，为主角、魔物、卡牌拖拽箭头、意图和伤害数字留出清晰阅读空间。
- 当前 MVP 主界面背景以白色纸面为底色，通过 `asset.background.mvp_battle.back` 与 `asset.background.mvp_battle.front` 两个层级叠放实现；前景层只承担地表和近景线条，不应烘焙 HUD、角色、魔物或机制反馈。主角脚底应落在前景地面线上。
- 战斗中的出牌失败提示和拖拽失败反馈不使用顶部横条提示框；优先表现为主角「左」的心理活动，使用从角色附近延伸出的逐渐变大的椭圆联想泡泡承载文字，持续约 2 秒后消失。普通回合流转提示不使用该泡泡，避免打断核心战斗节奏。

当前旧版战斗界面和资源围绕“8 格连锁点、3 / 5 / 8 阈值、行动牌 / 技能牌 / 终结牌”制作。当前三拍切片中应迁移为“6 格无色彩能槽 + 玩家拍位 + 魔物拍位 + 行动牌 / 终结牌”：

- 彩能 UI 使用 6 个基础槽位表达当前可存储彩能上限。
- 当前切片中每个彩能槽只需要显示无色彩能；后续五色阶段再显示无色、红、黄、蓝、绿、紫之一。
- 彩能槽 UI 不只是显示数量，还需要在后续五色阶段显示颜色构成；当前终结牌预览先明确本次会消耗多少无色彩能。
- 若后续效果突破 6 点上限，UI 应能显示额外彩能或溢出计数。
- 行动牌 hover 时，应展示本牌动作串、动作类型、数值、段数和成功产彩条件；附魔颜色和颜色追加效果后续接回。
- 终结牌 hover 时，应预览将消耗多少无色彩能、消耗后剩余彩能、基础效果和最终预估结果；每种被消耗颜色提供的追加效果后续接回。
- 最终预估结果应尽量列出最终伤害、护甲 / 防御、治疗、抽牌、异常、额外释放次数等玩家实际关心的收益。后续黄色附魔的预估结果只展示额外释放次数，不展示抽牌、返费或回能量。
- 后续五色阶段中，已附魔行动牌需要在卡牌上稳定显示色彩标记。

## 色彩预览与结算拆解（后续阶段）

色彩机制的 UI 目标不是只把牌染色，而是让玩家理解“这张牌的用途被颜色如何改写”。多色终结牌尤其需要清晰的结算拆解，避免玩家不知道为什么获得护甲 / 回复 / 额外弹幕。

行动牌预览至少包含：

- 基础费用、目标规则和基础效果。
- 当前附魔颜色；未附魔时显示为无色或未附魔状态。
- 打出后生成的彩能颜色与数量。
- 颜色对本牌本次结算的额外效果，例如红色格挡造成转化伤害、蓝色攻击获得护甲 / 防御、绿色控制附带回复。

终结牌预览至少包含：

- 本牌彩能需求：固定消耗、X 消耗或全部消耗。
- 将消耗多少彩能，以及将消耗哪些颜色。
- 基础效果。
- 每种被消耗颜色提供的追加效果，可按颜色分行显示。
- 同色叠加次数、当前上限和已触发次数。
- 最终预估收益，例如最终伤害、护甲 / 防御、治疗、抽牌、返费、异常、额外释放次数。

结算表现要求：

- 多色终结牌应先展示基础效果，再用短节奏追加展示颜色转译收益。
- 蓝色、绿色这类从最终伤害或最终效果值转译出的护甲 / 防御、治疗，需要在日志中标注来源。
- 黄色、紫色等高复杂触发应提供简短动画节奏和日志提示，避免连续触发时玩家丢失因果。
- 触发过多时，动画应合并相同颜色的小额收益，日志保留可展开或最近几条摘要。

## 战斗交互

- 行动牌交互采用“拖到己方拍位 -> 自动瞄准”的两步式连续操作：悬停卡牌时抬升并置顶，不触发布牌；拖到空玩家拍位松开后才请求放入。
- 放入成功后，该玩家拍位立即作为临时锚点拉出箭头；玩家点击合法魔物拍位或空门本体完成目标锁定。
- 临时瞄准期间，右键或点击空白区域取消本次瞄准，刚放入的行动牌退回手牌，拍位清空。
- 手牌不允许直接拖到魔物拍位或魔物本体。
- 终结牌放入终结槽，不作为普通拍位行动牌打出；完整终结槽拖拽与漫画分镜演出继续迭代。
- 表现层只负责 hover、drag、箭头、拍位命中、目标高亮和预览展示；拍位、彩能、目标合法性和胜负仍由规则层校验。

## 战后奖励与附魔界面

普通战斗后奖励界面长期方向分为两个核心动作；当前三拍切片可先只实现武器卡牌三选一：

1. 色彩碎片：后续五色阶段展示本次获得的色彩碎片，并让玩家选择 1 张未附魔行动牌进行附魔。若没有可附魔行动牌，需要显示清楚的替代规则或暂存规则。
2. 武器卡牌三选一：展示 3 张来自当前主手 / 副手武器卡池的候选卡牌，玩家选择 1 张加入卡组。

候选卡牌必须清楚展示所属武器、卡牌类型、拍位占用、动作串、彩能生成条件、终结牌彩能消耗和稀有度。色彩互动后续接回。

精英战斗奖励界面在上述流程后展示普通遗物。Boss 战斗结算界面展示 MVP 通关，并提供简化 Boss 奖励，例如 Boss 遗物、Boss 特有卡牌或色彩核心。

## 起始武器界面

MVP 起始角色固定为「左」，但主副武器不固定。进入 Run 前应展示左轮剑和机械臂，并让玩家选择其中一个作为主手、另一个作为副手。确认后系统自动组成 8 张初始牌：

- 主手武器：固定初始池 4 张全部加入。
- 副手武器：固定初始池 4 张全部加入。

界面应显示当前主副手选择、两把武器的固定初始池摘要、主副手进入初始卡组的规则、武器定位和奖励池方向。第一版 MVP 需要实现主副武器选择，但不再要求玩家从起始卡池中自由选择卡牌；卡池相关实现可保留，作为未来拓展为起始构筑选择的基础。

## 交互准则

- 卡牌目标、拍位已占用、彩能不足、无法瞄准、无法结算原因必须即时反馈。
- 当前彩能数量、玩家拍位、魔物拍位锁定状态、下一张行动牌动作串和终结牌预计消耗必须即时反馈。
- 终结牌触发时需要清楚展示关键数值变化，例如伤害、倍率、抽牌数、状态层数和后续色彩附加效果。
- 当后续五色阶段终结牌因当前彩能颜色构成获得额外效果时，卡牌本体需要有明显但不过度干扰的高亮状态。
- 轮末前需要提示彩能将归零，除非当前遗物或卡牌效果会保留部分或全部彩能。
- 魔物意图和状态说明应优先通过悬停查看；点击魔物不得改变战斗目标选择状态。
- 关键决策需要确认，普通重复操作不应频繁打断。
- UI 文案优先清晰，少用只服务氛围的模糊表达。

## 待更新资源

以下旧资源 / 规范需要迁移：

- 8 格连锁 UI 改为 6 格基础彩能槽。
- 3 / 5 / 8 阈值提示移除。
- 技能牌模板移除或改造为行动牌的防御 / 控制子风格。
- 卡牌包奖励界面当前改为武器卡牌三选一；后续五色阶段再接回色彩碎片。
- UI 图标当前需要新增拍位、目标指向、无色彩能、终结槽和武器图标；五色、附魔、色彩核心图标留到后续阶段。
